# 案例 03 数据集设计：零售经营产品方案（综合闭环）

**文件**：
- `dataset/order_data.csv`（真实基座 UCI Online Retail II + 本地化改写 + 合成叠加，同案例01 口径）
- `dataset/reference_data_analysis/2b-real_rfm.csv`（**真实基座派生**：UCI 客户级 RFM 真算，1665 位客户，经 `/api/rfm` 消费）

**场景**：把「早会异常 → 驾驶舱洞察 → RFM 会员分层 → 改进动作 → 验收」端到端串成一份可交付的经营方案。

## 独有真实底座：客户级 RFM（这是案例03 区别于案例01 的地方）
| 列 | 义 | 性质 |
|---|---|---|
| 客户号 | 匿名 CustomerID | 真实（匿名） |
| 最近购买天数 R / 购买次数 F / 总消费 M | 由真实交易记录真算 | 真实 |
| 分层 | 重要价值 / 高价值流失 / 一般保持 / 流失预警 / 普通 | **规则派生·非事实标签** |

「**高价值流失**」群（M 高、R 大 = 消费高却久未回购，123 位/7.4%）经 `retailRfm()` 真实计算，是会员经营真正要抢救的抓手——别被「一般保持/普通」的大盘 base-rate 带偏。

## 说明（红线）
order_data 的真实 / 本地化改写 / 教学合成三类与案例01 同口径显式区分；RFM 分层为分位规则派生、非平台事实标签。**绝不把真实基座说成「确定性合成」，也绝不把规则派生说成事实标签**——两个方向的红线都要守。
