# 第一部分 · 系统设计理念与原理

## 1. AI 核心概念底层

![AI 核心概念底层链路](outputs/product_case_library/svg/fig_ai_foundations.svg)

> 从底层到上层拆解 AI 核心概念，搭建扎实的认知框架（据「LLM、Token 等 AI 核心概念底层工程解析」）。本章工具无关，只讲原理。

### 1.1 大语言模型
#### 1.1.1 自回归
#### 1.1.2 概率预测
#### 1.1.3 结束标识

```备注
LLM（大语言模型）是生成式 AI 的核心底层基础，绝大多数大模型基于 Transformer 架构（Google 2017《Attention Is All You Need》）训练。其底层逻辑是「自回归文字接龙」：接收指令后预测概率最高的首个词元，把新生成的词汇追加到输入后再预测下一个，逐词迭代，直到输出特殊结束标识符。这就是为什么大模型永远逐词、逐字输出答案。
```

### 1.2 Token 与分词
#### 1.2.1 编码切分
#### 1.2.2 映射编号
#### 1.2.3 量化换算

```备注
大模型内部只识别数字、读不懂文字，人机之间靠 Tokenizer（分词器）翻译：编码=切分（把文本拆成最小片段 Token）+ 映射（每个 Token 绑定一个 Token ID 数字）；解码=反向映射。核心误区：Token ≠ 词语，切分逻辑基于训练语料。量化标准：1 Token ≈ 0.75 个英文单词 ≈ 1.5~2 个汉字；40 万 Token ≈ 60~80 万汉字。Token 是算力与上下文长度的计量单位。
```

### 1.3 上下文与窗口
#### 1.3.1 上下文
#### 1.3.2 上下文窗口
#### 1.3.3 检索增强

```备注
大模型本身只是「输入对应输出」的数学函数、无真实记忆；能「记住」聊天，是因为每次都把完整历史对话拼接到当前提问一起传入。Context（上下文）= 单次运算接收的全部输入信息总和，是临时记忆载体（含历史对话、系统指令、工具数据等）。Context Window（上下文窗口）= 单次能承载的最大 Token 数，决定记忆长度（主流已达百万级）。超长文档不应全量塞入，而用 RAG 检索增强：只把与问题高度匹配的核心片段 + 问题传入，降本、聚焦、更精准。
```

### 1.4 提示词
#### 1.4.1 用户提示
#### 1.4.2 系统提示
#### 1.4.3 提示词工程

```备注
Prompt（提示词）是驱动模型输出的具体指令，其精准度直接决定输出质量。分两类各司其职：User Prompt（用户提示词）由用户输入，定任务、定需求、定内容；System Prompt（系统提示词）由开发者后台预设、用户不可改，定人设、定规则、定输出边界，全程贯穿对话。提示词工程的本质是精准、清晰、完整地描述任务，消除指令模糊性——正是本教程 Prompt 实操的底层依据。
```

### 1.5 工具与协议
#### 1.5.1 工具
#### 1.5.2 四方协作
#### 1.5.3 统一接入

```备注
大模型天然无法感知、触达外部真实环境。Tool（工具）是可被调用的函数：接收指定入参，经内部逻辑或外部接口返回结构化结果，补齐外部感知短板。工具调用涉及用户、平台、大模型、工具四方：模型判定需调用并生成「工具调用指令」，平台解析并执行、把结构化结果回传模型整理成自然语言。MCP（模型上下文协议）是一套通用的工具统一接入标准（类比 Type-C），让工具「一次开发、多处复用」。
```

### 1.6 智能体与技能
#### 1.6.1 智能体
#### 1.6.2 循环推理
#### 1.6.3 智能体技能

```备注
Agent（智能体）具备自主规划、分步推理、循环调用工具的能力，让 AI 从被动应答升级为主动拆解复杂任务、自主迭代直至完成——其底层就是「思考→调用工具→获取结果→再思考」的 ReAct 循环推理架构（这正与第 2 章的三层 Loop 相接）。Agent Skill（智能体技能）是给智能体固化的专属执行手册：结构化写清触发条件、执行步骤、判断规则、输出格式，实现个性化能力一键复用、并省 Token。本教程的 Skill 库即此类结构化技能。
```

## 2. 理念框架（会 Loop 的产品工程）

![三层 Loop](outputs/product_case_library/svg/fig_ideology_loops.svg)

### 2.1 三层 Loop
#### 2.1.1 内层执行
#### 2.1.2 中层判断
#### 2.1.3 外层校准

```备注
AI 能自主写代码后，真正变化的不是效率，而是软件开发变成三种时间尺度的 Loop 同时运转。内层 Agentic Coding Loop：给 Spec 与 Evals，Agent 自己写码、跑测、发现问题继续改，分钟级自我纠偏。中层 Developer Feedback Loop：人不能离开，负责判断——功能范围、交互、信息组织，看完第一版再回改 Spec，几十分钟到小时级。外层 External Feedback Loop：真实用户与市场校准开发者的 Vision，几天到几周。一层比一层慢，也一层比一层重要；少任何一层，0→1 产品都难跑起来。稀缺能力正从内层向外层迁移。
```

### 2.2 提示词到 Loop
#### 2.2.1 提示词
#### 2.2.2 上下文
#### 2.2.3 脚手架
#### 2.2.4 循环

```备注
演进链路：Prompt Engineering（一条消息预见一切，靠措辞）→ Context Engineering（精心构造模型"看到"的：system prompt、few-shot、RAG、结构化输入，不再猜）→ Harness Engineering（给 Agent 装工具+重试+权限，一次 session 内多步执行）→ Loop Engineering（把"人触发→人判断→人决定下一步"里的人替换成能定时触发、验证、记忆、决定继续或停止的自动化系统）。与 cron 的区别：Loop 内部决策者是 LLM，行为在运行时才确定。
```

### 2.3 控制论闭环
#### 2.3.1 控制器
#### 2.3.2 执行器
#### 2.3.3 传感器

```备注
Loop 的本质是控制论：目标→执行→测量偏差→反馈修正→再测量。控制器（编排逻辑：读偏差、判断、构造下一轮 prompt）需要知识与记忆（→ Skills、Memory）；执行器（工具调用：改文件/命令/API）需要触达与隔离（→ Connectors、Worktrees）；传感器（验证，产出误差信号）需要独立性（→ Sub-agents）。收敛速度由传感器决定：只回 pass/fail 近乎盲猜，回"哪个用例挂、哪个断言失败、哪个 diff 引入"才是针对缺陷修复。**传感器就是设计本身**——"好提示+弱验证"会失败，"平庸提示+强验证"会收敛。
```

### 2.4 Loop 六件套
#### 2.4.1 触发
#### 2.4.2 隔离
#### 2.4.3 技能
#### 2.4.4 连接
#### 2.4.5 子代理
#### 2.4.6 记忆

```备注
六个组件是控制自然需要的基础设施：Automations（触发：定时/事件/心跳，谁按开始键）；Worktrees（隔离：每 Agent 一份独立工作区，并行不打架）；Skills（技能：把项目约定/踩坑写下来，每轮直接读，省 token）；Connectors（连接：MCP 接 GitHub/工单/通知，触达真实工具链）；Sub-agents（子代理：写的与验的分开，写代码的模型给自己打分太宽容）；Memory/State（记忆：上下文窗口每轮清空，磁盘存"做完了什么、下一步干啥"）。
```

### 2.5 何时建 Loop
#### 2.5.1 可重复
#### 2.5.2 可判定
#### 2.5.3 预算够
#### 2.5.4 可复现

```备注
Loop 不免费（烧 token、要搭、出问题要 debug 你没盯着的系统）。动手前四问：任务是否重复（一次性活好 prompt 更划算）？有没有东西能自动判定"干砸了"（测试/类型/lint/构建）？token 预算扛得住浪费？Agent 跑得了自己写的代码（有日志、能复现、看得到哪崩）？附加题最重要：你打算 review 产出吗——不打算就别建。目标不可测量的任务（"更优雅""更好"）不要建 Loop，那是假闭环。先建 Inspector，再建 Loop。
```

### 2.6 AI 软件工程框架
#### 2.6.1 强制方法论
#### 2.6.2 上下文管理
#### 2.6.3 虚拟团队
#### 2.6.4 仓库记忆

```备注
从"AI 写代码"到"AI 做软件工程"的工程化拐点——竞争点不是模型能力，是工程化标准。四类代表（工具无关的理念/开发模式，可套用于任意 Agent 工具）：Superpowers=强制方法论（brainstorm→writing-plans→TDD→code-review→subagent-driven，规则而非建议）；GSD=轻量 context 管理（解决 context rot，spec 驱动，.planning/.memory/.context）；GStack=虚拟工程团队（office-hours/CEO/Eng/Design/Staff/QA/Ship 角色分工，一个人跑完一个团队流程）；Trellis=仓库级记忆（Plan→Implement→Verify→Finish 四阶段，把规范/任务/记忆沉淀进仓库、每次 Finish 都为下次积累）。可组合：强制方法论+仓库记忆、轻量 context+虚拟团队。
```

### 2.7 产品视角总纲
#### 2.7.1 上下文优势
#### 2.7.2 判断力
#### 2.7.3 演示验证

```备注
人类的优势与其说是"品味"，不如说是上下文优势：用户是谁、业务边界、约束条件、竞品在做什么——这些还锁在人脑里。只要人还掌握 Agent 不知道的上下文，就必须留在 Loop 里把信息补给系统。Loop 放大的是你的判断力：你对系统理解多深，传感器就能写多精确。本知识库据此组织——先讲理念、原理、工程约束与设计，再用不同案例来演示、验证；每个案例都标注它演示了哪些原理、采用了哪套设计。
```

## 3. 系统架构设计（产品视角）

![产品视角架构设计流程](outputs/product_case_library/svg/fig_arch_flow.svg)

> 同一套方法论：从产品与业务出发，走完系统架构设计。架构不是技术选型清单，而是在特定约束下对质量属性做出的一组可追溯决策。（复用并优化自本仓库早期架构教程。）

### 3.1 约束先行
#### 3.1.1 业务约束
#### 3.1.2 合规约束
#### 3.1.3 规模约束

```备注
架构的起点不是技术，是约束。先把产品/业务的硬约束列清：业务边界与干系人诉求、合规与安全红线（如等保/日志留存/隐私）、规模与增长（并发、数据量、峰值）。约束表是后续每个决策的作证依据——没有约束支撑的复杂度，就是没有证据的复杂度。
```

### 3.2 质量属性
#### 3.2.1 性能场景
#### 3.2.2 可用性
#### 3.2.3 安全

```备注
把"非功能需求"写成可量化的质量属性场景（SEI 风格：刺激→环境→响应→度量），如"峰值 400 点/秒写入，P95<200ms"。两个功能清单相同的系统，可因质量属性优先级不同而导出完全不同的架构。质量属性场景是拆分与选型的证据来源。
```

### 3.3 子系统分解
#### 3.3.1 按数据流
#### 3.3.2 按变更率
#### 3.3.3 模块边界

```备注
拆分要有过硬证据：按数据流（写路径 vs 读路径）、按变更节奏（稳定的与频繁迭代的）、按故障与发布边界切分。模块化单体优先——模块边界就是预留的进程边界，"预留而不预付"。用架构守护测试把边界变成可执行的适应度函数，架构决策才不只是墙上的海报。
```

### 3.4 接口契约
#### 3.4.1 错误信封
#### 3.4.2 幂等
#### 3.4.3 契约即代码

```备注
接口是子系统间的合同：统一错误信封 {code,message,details}、写操作幂等、404 不泄露资源存在性、分页与 allowedActions 由服务端驱动。契约即代码——OpenAPI 由路由 schema 自动生成，契约与实现同源不漂移。
```

### 3.5 ADR 决策
#### 3.5.1 背景
#### 3.5.2 决策
#### 3.5.3 后果

```备注
每个关键决策写一篇 ADR（Nygard 格式）：背景（约束与证据）、决策（选了什么）、后果（正负影响、演进触发条件）。重点是"为什么这样选"，否则几周后仍会重复争论。决策可追溯，是产品视角架构的核心产物。
```

### 3.6 部署演进
#### 3.6.1 部署视图
#### 3.6.2 演进触发
#### 3.6.3 奥卡姆

```备注
部署图画的是约束表描述的真实机房，不是理想拓扑。用演进触发表把"何时该改造"写成可观察条件（规模×N、新需求立项…）提前量。奥卡姆原则贯穿：能用一张聚合表解决就不上专用集群，复杂度花在业务上，每个组件都要有证据。
```

## 4. 工程规范与代码约束

![工程规范与代码约束](outputs/product_case_library/svg/fig_engineering_rules.svg)

> 既是本教程的工程原理，也真实约束本仓库代码，并被 `verify_course_package.mjs` 守护。全文规范沉淀在 `rules/`（对齐 Google Style Guides、swe-book、Conventional Commits、OWASP 等，见 `rules/references.md`）。

### 4.1 复用优先
#### 4.1.1 成熟方案
#### 4.1.2 避免重复
#### 4.1.3 最小改动

```备注
优先调研成熟稳定的开源库，不重复造轮子；两处以上可复用逻辑抽离为公共模块（DRY）；优先在现有代码上扩展而非推倒重写、保持 API 兼容、改动范围最小化。见 rules/ai-dev-constraints.md 第 1/4/10 条。
```

### 4.2 结构清晰
#### 4.2.1 组件拆分
#### 4.2.2 目录规范
#### 4.2.3 分层架构

```备注
按功能拆分、单一职责，页面组织业务而不实现所有细节；新增代码遵循既定目录、不随意加顶级目录、工具/组件/接口/类型不混放；后端分层 controller→service，controller 不写业务、service 不碰 HTTP 上下文。见 rules/ai-dev-constraints.md 第 2/3 条与 rules/backend.md。
```

### 4.3 代码质量
#### 4.3.1 简洁<800行
#### 4.3.2 类型安全
#### 4.3.3 中文注释

```备注
可读性优先于炫技，一个函数只做一件事，不留废弃代码/调试输出；**单文件 < 800 行**，超出即拆分；优先 TS 类型、避免大量 any；统一中文注释，重点写"为什么这样做"与业务规则，复杂函数（约 30 行以上）前加整体思路。见 rules/ai-dev-constraints.md 第 5/6/7 条。
```

### 4.4 健壮与安全
#### 4.4.1 错误处理
#### 4.4.2 性能意识
#### 4.4.3 安全红线

```备注
所有异步请求做异常处理、不忽略 Promise 错误、用户可感知错误给明确提示；按需加载、避免不必要重复渲染；输入校验、防路径遍历、不硬编码密钥、敏感配置走环境变量。见 rules/ai-dev-constraints.md 第 8/9 条与 rules/backend.md 安全节（OWASP Top 10）。
```

### 4.5 前端规范
#### 4.5.1 Monorepo
#### 4.5.2 微前端
#### 4.5.3 公共沉淀

```备注
Monorepo（pnpm workspace）+ 微前端（qiankun）：shell 只做入口不写业务、业务放各子应用；公共能力沉淀 packages/（ui/theme/auth/request/utils），不重复封装请求/认证/主题；VITE_ 环境变量、独立构建。见 rules/frontend.md。
```

### 4.6 后端规范
#### 4.6.1 统一响应
#### 4.6.2 结构化日志
#### 4.6.3 优雅退出

```备注
统一响应 {code,message,data}、RESTful、按模块分组路由；结构化日志（slog）禁 printf 风格、语义化状态码；长任务用协程+channel、退出时清理资源、WebSocket 心跳超时。见 rules/backend.md。
```
