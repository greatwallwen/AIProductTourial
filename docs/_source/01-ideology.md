### 1.1 三层 Loop
#### 1.1.1 内层执行
#### 1.1.2 中层判断
#### 1.1.3 外层校准

```备注
AI 能自主写代码后，真正变化的不是效率，而是软件开发变成三种时间尺度的 Loop 同时运转。内层 Agentic Coding Loop：给 Spec 与 Evals，Agent 自己写码、跑测、发现问题继续改，分钟级自我纠偏。中层 Developer Feedback Loop：人不能离开，负责判断——功能范围、交互、信息组织，看完第一版再回改 Spec，几十分钟到小时级。外层 External Feedback Loop：真实用户与市场校准开发者的 Vision，几天到几周。一层比一层慢，也一层比一层重要；少任何一层，0→1 产品都难跑起来。稀缺能力正从内层向外层迁移。
```

### 1.2 提示词到 Loop
#### 1.2.1 提示词
#### 1.2.2 上下文
#### 1.2.3 脚手架
#### 1.2.4 循环

```备注
演进链路：Prompt Engineering（一条消息预见一切，靠措辞）→ Context Engineering（精心构造模型"看到"的：system prompt、few-shot、RAG、结构化输入，不再猜）→ Harness Engineering（给 Agent 装工具+重试+权限，一次 session 内多步执行）→ Loop Engineering（把"人触发→人判断→人决定下一步"里的人替换成能定时触发、验证、记忆、决定继续或停止的自动化系统）。与 cron 的区别：Loop 内部决策者是 LLM，行为在运行时才确定。
```

### 1.3 控制论闭环
#### 1.3.1 控制器
#### 1.3.2 执行器
#### 1.3.3 传感器

```备注
Loop 的本质是控制论：目标→执行→测量偏差→反馈修正→再测量。控制器（编排逻辑：读偏差、判断、构造下一轮 prompt）需要知识与记忆（→ Skills、Memory）；执行器（工具调用：改文件/命令/API）需要触达与隔离（→ Connectors、Worktrees）；传感器（验证，产出误差信号）需要独立性（→ Sub-agents）。收敛速度由传感器决定：只回 pass/fail 近乎盲猜，回"哪个用例挂、哪个断言失败、哪个 diff 引入"才是针对缺陷修复。**传感器就是设计本身**——"好提示+弱验证"会失败，"平庸提示+强验证"会收敛。
```

### 1.4 Loop 六件套
#### 1.4.1 触发
#### 1.4.2 隔离
#### 1.4.3 技能
#### 1.4.4 连接
#### 1.4.5 子代理
#### 1.4.6 记忆

```备注
六个组件是控制自然需要的基础设施：Automations（触发：定时/事件/心跳，谁按开始键）；Worktrees（隔离：每 Agent 一份独立工作区，并行不打架）；Skills（技能：把项目约定/踩坑写下来，每轮直接读，省 token）；Connectors（连接：MCP 接 GitHub/工单/通知，触达真实工具链）；Sub-agents（子代理：写的与验的分开，写代码的模型给自己打分太宽容）；Memory/State（记忆：上下文窗口每轮清空，磁盘存"做完了什么、下一步干啥"）。
```

### 1.5 何时建 Loop
#### 1.5.1 可重复
#### 1.5.2 可判定
#### 1.5.3 预算够
#### 1.5.4 可复现

```备注
Loop 不免费（烧 token、要搭、出问题要 debug 你没盯着的系统）。动手前四问：任务是否重复（一次性活好 prompt 更划算）？有没有东西能自动判定"干砸了"（测试/类型/lint/构建）？token 预算扛得住浪费？Agent 跑得了自己写的代码（有日志、能复现、看得到哪崩）？附加题最重要：你打算 review 产出吗——不打算就别建。目标不可测量的任务（"更优雅""更好"）不要建 Loop，那是假闭环。先建 Inspector，再建 Loop。
```

### 1.6 AI 软件工程框架
#### 1.6.1 强制方法论
#### 1.6.2 上下文管理
#### 1.6.3 虚拟团队
#### 1.6.4 仓库记忆

```备注
从"AI 写代码"到"AI 做软件工程"的工程化拐点——竞争点不是模型能力，是工程化标准。四类代表（工具无关的理念/开发模式，可套用于任意 Agent 工具）：Superpowers=强制方法论（brainstorm→writing-plans→TDD→code-review→subagent-driven，规则而非建议）；GSD=轻量 context 管理（解决 context rot，spec 驱动，.planning/.memory/.context）；GStack=虚拟工程团队（office-hours/CEO/Eng/Design/Staff/QA/Ship 角色分工，一个人跑完一个团队流程）；Trellis=仓库级记忆（Plan→Implement→Verify→Finish 四阶段，把规范/任务/记忆沉淀进仓库、每次 Finish 都为下次积累）。可组合：强制方法论+仓库记忆、轻量 context+虚拟团队。
```

### 1.7 产品视角总纲
#### 1.7.1 上下文优势
#### 1.7.2 判断力
#### 1.7.3 演示验证

```备注
人类的优势与其说是"品味"，不如说是上下文优势：用户是谁、业务边界、约束条件、竞品在做什么——这些还锁在人脑里。只要人还掌握 Agent 不知道的上下文，就必须留在 Loop 里把信息补给系统。Loop 放大的是你的判断力：你对系统理解多深，传感器就能写多精确。本知识库据此组织——先讲理念、原理、工程约束与设计，再用不同案例来演示、验证；每个案例都标注它演示了哪些原理、采用了哪套设计。
```
