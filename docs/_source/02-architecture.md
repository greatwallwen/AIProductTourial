## 3. 系统架构设计（产品视角）

![产品视角架构设计流程](outputs/product_case_library/svg/fig_arch_flow.svg)

> 同一套方法论：从产品与业务出发，走完系统架构设计。架构不是技术选型清单，而是在特定约束下对质量属性做出的一组可追溯决策。（复用并优化自本仓库早期架构教程。）

### 3.1 约束先行
#### 3.1.1 业务约束
#### 3.1.2 合规约束
#### 3.1.3 规模约束

```备注
新手做架构，第一反应往往是「用什么技术栈、上不上微服务、要不要上 Kafka」。但你有没有发现，脱离场景聊技术选型，永远吵不出结果——因为**架构的起点根本不是技术，是约束**。

先把产品和业务的硬约束一条条列清楚：业务边界和干系人到底要什么、有哪些合规与安全红线（等保、日志留存、隐私）、规模会长到多大（并发、数据量、峰值）。这张「约束表」，是你后面每一个架构决策的**证据**。

一句话点破：**没有约束支撑的复杂度，就是没有证据的复杂度**。你上了一套微服务，若说不出是哪条约束逼你上的，那它多半是负债，不是架构。
```

### 3.2 质量属性
#### 3.2.1 性能场景
#### 3.2.2 可用性
#### 3.2.3 安全

```备注
举个最能说明问题的例子：两个系统，功能清单一模一样——都是「用户下单、查订单、退款」。它们的架构会一样吗？不一定，甚至可能天差地别。为什么？因为**质量属性**不同。

一个是电商大促系统，要扛「峰值 400 单/秒、P95 延迟 <200ms」；另一个是企业内部系统，日均几百单但要求「7×24 可用、任何故障 5 分钟内可回滚」。前者会为吞吐做读写分离、缓存、异步；后者会为可用性做冗余、灰度、快速回滚。功能相同，质量属性优先级不同，导出的架构就完全不同。

所以架构师的功夫，是把模糊的「非功能需求」写成**可量化的质量属性场景**（SEI 风格：刺激→环境→响应→度量，如上面那两句）。质量属性场景，就是你后面做拆分和选型的证据来源——没有它，选型就是拍脑袋。
```

### 3.3 子系统分解
#### 3.3.1 按数据流
#### 3.3.2 按变更率
#### 3.3.3 模块边界

```备注
拆分要有过硬证据：按数据流（写路径 vs 读路径）、按变更节奏（稳定的与频繁迭代的）、按故障与发布边界切分。模块化单体优先——模块边界就是预留的进程边界，"预留而不预付"。用架构守护测试把边界变成可执行的适应度函数，架构决策才不只是墙上的海报。
```

### 3.4 接口契约
#### 3.4.1 错误信封
#### 3.4.2 幂等
#### 3.4.3 契约即代码

```备注
接口是子系统间的合同：统一错误信封 {code,message,details}、写操作幂等、404 不泄露资源存在性、分页与 allowedActions 由服务端驱动。契约即代码——OpenAPI 由路由 schema 自动生成，契约与实现同源不漂移。
```

### 3.5 ADR 决策
#### 3.5.1 背景
#### 3.5.2 决策
#### 3.5.3 后果

```备注
每个关键决策写一篇 ADR（Nygard 格式）：背景（约束与证据）、决策（选了什么）、后果（正负影响、演进触发条件）。重点是"为什么这样选"，否则几周后仍会重复争论。决策可追溯，是产品视角架构的核心产物。
```

#### 3.6.1 部署视图
#### 3.6.2 演进触发

```备注
部署图画的是约束表描述的真实机房，不是理想拓扑。用演进触发表把"何时该改造"写成可观察条件（规模×N、新需求立项…）提前量。奥卡姆原则贯穿：能用一张聚合表解决就不上专用集群，复杂度花在业务上，每个组件都要有证据。
```
