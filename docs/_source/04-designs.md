## 附录A · 设计系统（多套，各不相同）

![多套设计系统](outputs/product_case_library/svg/fig_designs.svg)

> 不是一套统一审美：`design/` 下 ≥5 套**各不相同**的深色大屏设计系统（色板/布局/气质各异，对齐大屏参考与 voltagent DESIGN.md 模型）。不同案例采用不同设计 → 大屏风格各异。机器令牌单一来源 `design/themes.json`，人读文档 `design/<id>.md`。

> 🎯 **本章学习目标**（读完你能——）
> - 理解「设计系统」为什么不是配色偏好，而是让不同案例**风格各异却各自专业**的一套令牌；
> - 认得五套深色大屏设计（HUD/流向/漏斗/矩阵/队列）各适配什么业务场景；
> - 知道「机器令牌单一来源 + 人读文档」如何保证设计一致、可维护。
>
> 🧭 **难度** ★★☆ 进阶 ｜ **前置** 无 ｜ **预计** 12 分钟 ｜ 每套设计对应到不同案例，看案例时可回查本章。

### A.1 石墨 HUD
> 🟢 **必读** ｜ ★☆☆ ｜ 关键词：**青蓝高光** · **上图下表** · **经营驾驶舱**（案例 01/41 用它）

```备注
为什么同样是「深色大屏」，有的看着高级、有的像套了模板的 PPT？差别就在设计系统。石墨 HUD 这套：石墨黑打底、青蓝(#22d3ee)做高光、铺一层 HUD 网格线，布局走「顶部总览 → 中部趋势主图 → 底部明细表」的经典驾驶舱结构——信息密度拉满但不乱。它适合 executive_dashboard、hr 这类要一眼看全局的经营驾驶舱（本教程电商早会、零售方案就用它）。完整色板/字号/圆角令牌见 design/graphite-hud.md。
```

### A.2 翡翠流向
> 🟢 **必读** ｜ ★☆☆ ｜ 关键词：**翡翠发光** · **桑基瀑布** · **流程流向**（案例 04/09 用它）

```备注
1898 年，爱尔兰工程师 Matthew Sankey 画了一张图，用一条条粗细不同的带子表示蒸汽机里能量「流」到哪、损耗在哪——这就是沿用至今的「桑基图」。它的精髓是：让**流动和转化**一眼可见。翡翠流向这套设计就是为这类场景生的：近黑墨绿打底、翡翠绿（#10b981）做发光高光，天生带一种「东西在往下淌」的流向感。

布局走桑基 / 瀑布式的纵向结构，适配 workflow_pipeline、knowledge 这类**流程流向、层层转化**的画面（本教程的 RAG 检索案例 04 与仓库事件总线案例 09 都用它——数据从语料一路「流」到命中片段、事件从提交一路「流」成时间线）。完整色板令牌见 `design/emerald-flow.md`。
```

### A.3 琥珀漏斗
> 🟢 **必读** ｜ ★☆☆ ｜ 关键词：**暖金高光** · **漏斗级联** · **转化收束**（案例 02 用它）

```备注
1898 年（又是这一年，营销和可视化在同一时代萌芽），广告先驱 E. St. Elmo Lewis 提出了 AIDA 模型：注意→兴趣→欲望→行动，一层层筛下来——这就是所有「转化漏斗」的祖宗。漏斗的视觉语言天然是**上宽下窄、层层收束**，每一层都在问「这一步漏掉了多少人」。

琥珀漏斗这套设计正是为它调的色：深棕琥珀打底、暖金（#f59e0b）做高光，暖色调本身带一点「该关注流失」的警示感。布局走漏斗级联，适配 sales_funnel 这类**转化分析**场景（本教程大陆 P2P 信贷案例 02 就用它——借款人从「信用画像分层」到「放款转化」的层层收束，正是一只漏斗）。完整令牌见 `design/amber-funnel.md`。
```

### A.4 青紫矩阵
> 🟢 **必读** ｜ ★★☆ ｜ 关键词：**双色高光** · **矩阵热力** · **密集决策**（案例 06/07/08 用它）

```备注
1970 年，波士顿咨询的 Bruce Henderson 画出那张著名的「波士顿矩阵」——把业务按两个维度铺进四个格子（明星、金牛、问题、瘦狗），复杂的组合决策一下子有了坐标。这就是「矩阵思维」的力量：**当你要同时权衡两三个维度时，把它们铺成一张网格或热力图，离群和优先级自己就跳出来了。**

青紫矩阵这套设计为密集决策而生：墨蓝打底、青（#06b6d4）与品紫（#a855f7）双高光，信息密度高但靠双色区分层次。布局走矩阵 / 热力 / 散点，适配 matrix_decision、finance、风控复核这类**多维密集决策**场景（本教程架构契约案例 06、RAG 评测台案例 07、SDD 建造走查案例 08 都用它）。完整令牌见 `design/cyan-matrix.md`。
```

### A.5 钢灰队列
> 🟢 **必读** ｜ ★☆☆ ｜ 关键词：**冷蓝高光** · **复核队列** · **明细留痕**（案例 05 用它）

```备注
2014 年前后，Salesforce 的设计师 Jina Anne 把一个概念推向业界主流，叫 **Design Token（设计令牌）**：把颜色、字号、圆角这些最小设计决策抽成一份「变量清单」，代码和设计共用同一个源头，改一处、全局同步。本教程的 `design/themes.json` 就是这么一份令牌单一来源——五套设计之所以能「各不相同却各自自洽」，靠的正是它，而不是每个页面各调各的色。

钢灰队列这套令牌走冷静克制的路线：钢灰近黑打底、冷蓝（#60a5fa）高光，适合需要长时间盯着看、强调**明细与留痕**的场景。布局走复核队列 / 明细表，适配 review_queue、list_review、record_detail（本教程关系库查询 45 用它）。完整令牌见 `design/steel-queue.md`。
```

### A.6 反 AI-slop：给 AI 一套设计词汇表
> 🔵 **选读·进阶** ｜ ★★☆ ｜ 关键词：**AI slop** · **OKLCH** · **不嵌套卡片**（本书也在自查自己的 AI 味）

```备注
你让 AI 写个落地页，它交出来排版整齐、配色和谐，可做设计的朋友一眼就说「这明显是 AI 生成的」。为什么？因为所有前端大模型都啃过同一批素材，练出了一身「审美肌肉记忆」——紫色渐变背景、Inter 字体铺满、卡片里再套卡片、圆角图标叠在标题上、hero 上方一个药丸小标签、「Not a tool. A platform.」式三段对比文案。单看每条都没错，组合起来就是 **AI slop**（AI 生成的廉价指纹）。

开源技能 Impeccable（Apache-2.0）的解法不是给 AI 更长的 prompt，而是给它一套**结构化设计词汇表** + 一个纯规则、无需 API key 的检测器（约 45 条反模式规则）：禁用被用烂的 Inter、改用 OKLCH 色彩空间而非到处套模板色、不许卡片套卡片而用间距与层级、弹跳缓动换成指数衰减。它最狠的一条哲学叫 **practice what you preach**——它的官网必须能通过它自己的检测，否则整个产品的可信度就崩了。

这对本书是双重印证：一是 附录A 的 Design Token 单一来源，正是「用系统对抗随手拍脑袋」；二是本书正在做的**去 AI 味**（去套话、把 emoji 换成专业图标）——同一种自律：不光让 AI 少出 slop，更要让自己的产物经得起「这看着像不像 AI 写的」这一问。

去 AI 味如今也长出一批专门工具（humanizer / stop-slop / taste-skill / ai-flavor-remover / shuorenhua 一类「去味」skill）。本书 dogfood 的做法是：`skills/` 里备了 `humanizer` / `de-ai-flavor` 两个去味技能，再用 `adversarial_review` 的 drift 探针 + `verify` 的去味词表当门禁——文字一旦混进营销空话或翻译腔套话，当场报红，逼着改回人话。
```

### 本章小结

- **设计系统 ≠ 配色偏好**：它是一套「令牌 + 布局 + 气质」的规则，让不同案例风格各异，却每套都专业、可维护。
- **五套各有主场**：HUD 看全局（驾驶舱）、流向看转化路径（桑基）、漏斗看层层收束、矩阵看多维密集决策、队列看明细复核——先有业务场景，再选设计。
- **一致性靠单一来源**：`design/themes.json` 是机器令牌唯一源头（Design Token 思想），`design/<id>.md` 是人读文档，改一处全局同步。

### 练习

1. **巩固**：一个「广告投放转化复盘」大屏，你会选五套里的哪一套？为什么？
2. **挑战**：如果要新增第六套设计「用于医院号源调度大屏」，你会先改哪个文件、定义哪些令牌？说说它的色板气质应该偏冷还是偏暖，为什么。
