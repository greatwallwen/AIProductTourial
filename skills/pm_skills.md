## problem-framing

- 类型：构建型
- 触发条件：当需要把技术任务或现象重述为可验证的产品问题时
- 输入：原始诉求/现象描述 + 可用数据字段清单
- 澄清问题：这是谁的问题？痛在哪个业务动作上？不解决的代价是什么？
- PRD 片段：把技术任务或现象重述为可验证的产品问题，输出结构化交付物并标注数据来源
- 验收标准：问题一句话可证伪、绑定一个可计算指标、注明数据来源与边界
- 复用范围：1 电商早会异常订单台

## metric-definition

- 类型：构建型
- 触发条件：当需要为业务对象定义可计算、可回溯的指标链时
- 输入：业务目标 + 数据字段（含口径说明）
- 澄清问题：指标分子分母各取哪列？口径边界（退货/取消算不算）？异常如何进队列？
- PRD 片段：为业务对象定义可计算、可回溯的指标链，输出结构化交付物并标注数据来源
- 验收标准：每个指标给出计算式与来源列、能被脚本复算、有异常阈值
- 复用范围：1 电商早会异常订单台

## acceptance-criteria

- 类型：验收/守护型
- 触发条件：当需要把交付物写成可执行的验收条件清单时
- 输入：交付物草案 + 指标链
- 澄清问题：每条验收能否自动判定？谁签收？失败时的回退动作是什么？
- PRD 片段：把交付物写成可执行的验收条件清单，输出结构化交付物并标注数据来源
- 验收标准：条条可执行可判定、含负例、有签收人
- 复用范围：1 电商早会异常订单台

## rfm-segmentation

- 类型：构建型
- 触发条件：当需要用 RFM 对客户分层并给出分层动作时
- 输入：会员消费记录（R/F/M 三列或可派生列）
- 澄清问题：R/F/M 分箱边界怎么定？「高价值流失」怎么定义？各层动作是什么？
- PRD 片段：用 RFM 对客户分层并给出分层动作，输出结构化交付物并标注数据来源
- 验收标准：分层规则可复算、各层有动作与预算边界、高价值流失单列
- 复用范围：02 航空会员价值运营

## lifecycle-action

- 类型：构建型
- 触发条件：当需要按客户生命周期定义运营动作时
- 输入：分层结果 + 可用触点渠道清单
- 澄清问题：每层的下一个最优动作是什么？频控与打扰边界？效果怎么回收？
- PRD 片段：按客户生命周期定义运营动作，输出结构化交付物并标注数据来源
- 验收标准：层-动作-渠道-度量四件套齐全、有频控约束、效果可回收
- 复用范围：02 航空会员价值运营

## privacy-boundary

- 类型：构建型
- 触发条件：当需要标注个人数据使用边界与脱敏要求时
- 输入：数据字段清单 + 使用目的
- 澄清问题：哪些列是个人数据？用途是否超出收集目的？脱敏后还够用吗？
- PRD 片段：标注个人数据使用边界与脱敏要求，输出结构化交付物并标注数据来源
- 验收标准：个人数据列全标注、脱敏方案可执行、高影响动作保留人工复核
- 复用范围：02 航空会员价值运营

## eval-design

- 类型：验收/守护型
- 触发条件：当需要为 AI 输出设计可量化的评测方案时
- 输入：待评测能力 + 语料/样本池
- 澄清问题：金标问题从哪来？命中怎么判定？多少分算过、回归下限设在哪？
- PRD 片段：为 AI 输出设计可量化的评测方案，输出结构化交付物并标注数据来源
- 验收标准：评测集可复跑、判定规则写成代码、有基线分与回归门
- 复用范围：07 RAG 评测

## harness-builder

- 类型：构建型
- 触发条件：当需要搭建原型验收护栏，逐项核验字段与目标时
- 输入：目标任务 + 可自动判定的传感器清单
- 澄清问题：传感器先于 Loop 存在吗？停机规则是什么？记忆落在哪里？
- PRD 片段：搭建原型验收护栏，逐项核验字段与目标，输出结构化交付物并标注数据来源
- 验收标准：传感器可脚本执行、有停机规则、跑一轮能产出误差信号
- 复用范围：07 RAG 评测 / 08 SDD 走查

## capstone-product-flow

- 类型：构建型
- 触发条件：当需要端到端串起产品思考到交付的综合闭环时
- 输入：多案例产出 + 北极星指标
- 澄清问题：闭环从哪个信号开始？每步交接物是什么？哪里必须人工把关？
- PRD 片段：端到端串起产品思考到交付的综合闭环，输出结构化交付物并标注数据来源
- 验收标准：信号→分析→决策→执行→复盘链路完整、每步有工件
- 复用范围：03 综合闭环

## evidence-pack

- 类型：构建型
- 触发条件：当需要把结论打包成可追溯的证据集时
- 输入：决策结论 + 原始数据/图表
- 澄清问题：每个结论的证据在哪一行数据？找过反例吗？口径与他人一致吗？
- PRD 片段：把结论打包成可追溯的证据集，输出结构化交付物并标注数据来源
- 验收标准：结论-证据一一对应、附可复算脚本或查询、无孤证
- 复用范围：03 综合闭环

## traceability-check

- 类型：构建型
- 触发条件：当需要核查需求到交付的双向可追溯时
- 输入：交付物 + 需求/规格清单
- 澄清问题：每条需求落到哪个工件？工件回指哪条需求？孤儿与悬空各有多少？
- PRD 片段：核查需求到交付的双向可追溯，输出结构化交付物并标注数据来源
- 验收标准：需求↔工件双向可追踪、无孤儿工件、无悬空需求
- 复用范围：03 综合闭环 / 08 SDD 走查

## spec-authoring

- 类型：构建型
- 触发条件：当要建一个中大型系统、需先把「要什么/为什么」写成规格（SDD 第②步）时
- 输入：目标、用户与场景、真实数据/资料、业务与合规约束
- 澄清问题：这个系统的目标与边界是什么？输入输出与数据是什么？有哪些业务/合规/规模/成本约束？
- PRD 片段：把目标写成 spec.md——含目标、用户场景、输入输出、边界、前后置条件、约束表；只写 what/why，不写实现（来源：GitHub Spec Kit）
- 验收标准：规格可追溯到真实约束、无实现细节泄漏、每条需求可被验收，模糊处已标 [需澄清]
- 复用范围：08 SDD 系统建造走查

## requirement-clarify

- 类型：验收·守护型
- 触发条件：当规格里有模糊处、需人在关口逐条消歧（SDD 第③步）时
- 输入：spec.md 及其中标注的 [需澄清] 项
- 澄清问题：这条需求的隐含假设是什么？边界情况如何处理？谁来拍板？
- PRD 片段：把所有模糊处标成 [需澄清] 并逐条问人确认，消除「意图债务」（§2.9），不让 Agent 替人猜
- 验收标准：所有 [需澄清] 均有人工确认结论、无未决歧义进入下一步、决策留痕
- 复用范围：08 SDD 系统建造走查

## c4-modeling

- 类型：构建型
- 触发条件：当要把系统从上下文逐层拆到组件、画出可交流的架构图时
- 输入：确认后的规格、子系统清单、外部依赖
- 澄清问题：系统跟谁交互（Context）？内部有哪些可独立运行的容器（Container）？某容器内部有哪些组件（Component）？
- PRD 片段：产出 C4 四层图（上下文/容器/组件/代码），每层只画对应粒度，依赖用带箭头连线（来源：C4 模型，Simon Brown）
- 验收标准：容器只含可独立部署/运行之物、依赖方向清晰、不把类与函数混进容器图
- 复用范围：06 子系统分解与接口契约

## domain-decomposition

- 类型：构建型
- 触发条件：当要按业务域把系统切成限界上下文、划清边界与通用语言时
- 输入：业务域、干系人、既有子系统
- 澄清问题：这块业务归谁管？说的是不是同一套语言？上下游怎么用契约对接？
- PRD 片段：按业务域划分限界上下文，每个上下文有独立模型与通用语言，标注上游/下游关系（来源：DDD，Eric Evans）
- 验收标准：上下文边界与业务边界一致、每个上下文语言自洽、跨上下文只经契约
- 复用范围：06 子系统分解与接口契约

## interface-contract

- 类型：验收·守护型
- 触发条件：当子系统之间要定接口、约定错误信封与幂等时
- 输入：接口清单、数据模型、调用方
- 澄清问题：错误怎么统一返回？写操作幂等吗？分页与可用动作谁驱动？
- PRD 片段：约定统一错误信封 {code,message,details}、写操作幂等、404 不泄露存在性、契约即代码（OpenAPI 由 schema 自动生成，§3.4）
- 验收标准：契约与实现同源不漂移、错误信封统一、写操作可安全重试
- 复用范围：06 子系统分解与接口契约

## adr-authoring

- 类型：验收·守护型
- 触发条件：当有关键技术/架构选型、需把「为什么这样选」留痕时
- 输入：待决策项、约束表、候选方案
- 澄清问题：当时有哪些约束和证据？选了什么？代价与重估信号是什么？
- PRD 片段：写三段式 ADR——背景（约束/证据）→决策（选了什么）→后果（好处/代价/出现什么信号该重估），重点写「为什么」（来源：ADR，Michael Nygard）
- 验收标准：每个关键决策有一份 ADR、含重估信号、可被后人追溯不必重吵
- 复用范围：06 子系统分解与接口契约

## quality-attribute-scenario

- 类型：构建型
- 触发条件：当要把模糊的非功能需求写成可量化场景、作为选型证据时
- 输入：非功能需求（性能/可用/安全/可复现…）、真实约束
- 澄清问题：什么刺激？什么环境？期望什么响应？用什么度量？
- PRD 片段：把非功能需求写成 SEI 质量属性场景「刺激→环境→响应→度量」，作为拆分与选型的证据来源
- 验收标准：每条场景四要素齐全、度量可量化、能导出具体架构决策
- 复用范围：06 子系统分解与接口契约

## fitness-function

- 类型：验收·守护型
- 触发条件：当要把架构边界写成能自动跑的断言、防止模块边界被破坏时
- 输入：模块边界规则（如 A 不许直接调 B 内部）、目录/依赖结构
- 澄清问题：哪些边界最容易被偷偷破坏？循环依赖如何检测？越界如何报警？
- PRD 片段：把架构约束写成自动化断言（适应度函数/架构守护测试），扫依赖检测越界与循环依赖，纳入门禁
- 验收标准：断言可自动跑、越界即报红并定位、纳入 verify 门禁
- 复用范围：06 子系统分解与接口契约

## task-decomposition

- 类型：构建型
- 触发条件：当要把架构计划拆成可并行、可独立验收的原子任务（SDD 第⑤步）时
- 输入：plan.md、子系统清单、依赖关系
- 澄清问题：哪些任务可并行？每个任务的验收条件是什么？依赖谁？
- PRD 片段：把计划拆成 tasks.md——原子、可并行、每个标注依赖/验收条件/所属子系统，供逐任务跑 Loop
- 验收标准：任务原子且可独立验收、依赖清晰无环、可并行标注正确
- 复用范围：08 SDD 系统建造走查

## arch-review

- 类型：验收·守护型
- 触发条件：当一次架构设计要过评审门禁、检查工件一致性与覆盖缺口时
- 输入：spec/plan/tasks/ADR 工件、门禁清单
- 澄清问题：工件之间一致吗？有无未覆盖的需求？约束是否都有证据支撑？
- PRD 片段：逐项核查架构工件一致性（规格↔架构↔任务↔ADR）、覆盖缺口、约束-复杂度证据链，出评审结论（只读、不改）
- 验收标准：工件互相一致、无未覆盖需求、每个复杂度都指得出约束当证据
- 复用范围：08 SDD 系统建造走查

## humanizer

- 类型：验收·守护型
- 触发条件：当产出一眼像 AI 写的（营销套话 / 三段排比 / 过度破折号）、要去掉 AI 痕迹时
- 输入：待改写文本 + 目标读者与语气
- 澄清问题：读者是谁？要口语还是正式？哪些事实/数据必须保留？
- PRD 片段：去 AI 写作痕迹——删「赋能/闭环/抓手/保驾护航」类套话与「不是 X 而是 Y」三段排比、拆过长破折号句，保留事实不注水（引真实工具生态 humanizer/stop-slop/taste-skill）
- 验收标准：信息不丢、无 AI 套话词、句式自然、能过「像不像 AI 写的」自检
- 复用范围：全书去 AI 化 / 导读与结课文案

## de-ai-flavor

- 类型：验收·守护型
- 触发条件：中文文案有「AI 腔」（书面、堆词、不说人话）要改口语时
- 输入：中文待改写文本
- 澄清问题：这句核心想说什么？能不能一句大白话说清？哪些词是凑数的？
- PRD 片段：中文说人话改写——去「值得注意的是/综上所述/众所周知」、去凑数形容词、一句话一个意思，向科普口吻靠（引 shuorenhua/ai-flavor-remover 榜）
- 验收标准：无中文 AI 套话、无空话形容词、读起来像人在讲、事实数据不变
- 复用范围：全书科普备注 / 中文文案去味

## growth-marketing

- 类型：构建型
- 触发条件：当要给一个功能/产品做增长或营销方案（渠道/文案/转化）时
- 输入：目标用户、产品卖点、可用渠道与预算
- 澄清问题：北极星指标是什么？目标人群在哪？转化漏斗每步的钩子是什么？
- PRD 片段：产出增长/营销方案——定北极星指标、拆「获取→激活→留存→变现」漏斗、每步给可验收的动作与文案，标注渠道与预算边界（引 marketing-skills 榜）
- 验收标准：指标可量化、漏斗每步有动作与度量、文案不空话（过去味）、不越合规边界
- 复用范围：03 经营闭环

## requirement-grill

- 类型：构建型
- 触发条件：当 spec 尚不存在、要在动手前把目标/边界/反例逼问清楚时（spec 已存在的 [需澄清] 消歧交给 requirement-clarify）
- 输入：一句话诉求 + 你能拿到的任何背景
- 澄清问题：这次要达成的可观察结果是什么？谁会用、在什么场景下？什么情况算失败/不做？
- PRD 片段：反向面试——由 Agent 拷问提出者：目标→用户→边界→反例→约束逐层追问，产出可进入 SDD ②规格的问题定义（来源：Matt Pocock skills · grill-me，MIT，核实 2026-07；按本书六槽与案例锚重写，非原文）
- 验收标准：每个分支有明确答案或标 [需澄清] 移交；无一处「大概/都行」
- 复用范围：08 SDD 走查 ②规格前置（见 system-build.orchestrator 前置步）
- 执行协议：一次只问一个问题；顺序 目标→用户→边界→反例→约束；答不出→标 [需澄清] 移交 requirement-clarify；连续 3 问无新信息或问满预算即停。

## test-first

- 类型：构建型
- 触发条件：当要让 Agent 写实现代码、需要一个无歧义的「做对了」判据时
- 输入：目标行为的一句话描述 + 可运行的测试环境
- 澄清问题：这个行为的最小可断言输出是什么？确定性吗（同输入同输出）？失败长什么样？
- PRD 片段：红→绿→重构——先写会挂的测试并确认它挂，再写最小实现使其过，最后重构；测试先入库（来源：Matt Pocock skills · tdd，MIT，核实 2026-07；重写，本仓库标本=code/tools/diagram.test.mjs）
- 验收标准：测试先于实现提交；含至少一条确定性断言（如同输入同输出）；红→绿过程可复现
- 复用范围：08 SDD 走查（⑥实现步）/ §4.3 标本
- 执行协议：写测试→跑并确认失败→最小实现→跑至全绿→重构→再跑；任一步跳过即视为未完成。

## self-quiz

- 类型：教员型
- 触发条件：当读完一章/一案、想检验「是否真记住」而非「感觉看懂了」时
- 输入：刚学的章节/案例名
- 澄清问题：不看书能复述几个关键概念？哪题答错说明该回读哪节？
- PRD 片段：出 5-8 道检索式自测题（先自答、后判分），错题逐条回指章节；与 check_my_work.mjs 互补——它查交付物、本卡查记忆（来源：Matt Pocock /teach · aihero.dev，MIT，核实 2026-07；重写）
- 验收标准：每题有明确判分依据；错题给出回读位置；连续两题空白即停止并建议回读
- 复用范围：各章练习 / 99 结课自查
- 执行协议：一次一题；先让学员作答再给答案；错→记回读锚；两题连续空白→停，输出回读清单。

## review-triage

- 复用范围：self-evolve Loop / 08 SDD 走查（⑦门禁）
- 执行协议：先按严重度分组→逐条判目标相关→产出 本轮修复序列+降级留痕表；不允许无理由 wontfix。

## shadcn-component-ownership

- 输入：设计需求 + 目标框架（React/Next.js）+ 无障碍/定制/包体约束
- 澄清问题：这些组件要不要深度定制？升级破坏性能不能承受？无障碍谁负责？

- 验收标准：组件源码在仓内可改、无破坏性升级依赖；变体用 cva 收敛；无障碍原语未被绕过

## design-tokens

- 类型：构建型
- 触发条件：当颜色/字号/圆角/间距散落各处、主题切换或一致性出问题时
- 输入：视觉规范草案 + 现有硬编码样式清单
- 澄清问题：哪些是决策（令牌）哪些是用法？暗色/多主题怎么切？谁是单一真源？
- PRD 片段：把最小设计决策（颜色/字号/圆角/间距/阴影）抽成 CSS 变量单一真源（本仓库 `--accent/--panel` 与 shadcn `--primary/--radius` HSL 令牌并存），组件只引用令牌不写死值——改一处全局生效（来源：Design Token 社区实践 + shadcn 主题令牌，核实 2026-07）

- 复用范围：附录A 设计系统 · 5 套主题（design/themes.json）

## ui-ux-heuristics

- 类型：验收·守护型
- 触发条件：当界面要做专业走查、判「好不好用」而非「有没有」时
- 输入：待走查界面/原型 + 目标用户任务
- 澄清问题：视觉层级对不对？对比够不够？状态反馈及时吗？出错可恢复吗？
- PRD 片段：用 Nielsen 十大可用性启发 + 层级/对比/一致/即时反馈/减负 做走查，逐条记「符合/违反+证据+修法」，把「好不好用」变成可复核清单（来源：Nielsen Norman 可用性启发，核实 2026-07；对齐附录A 反 AI-slop 立场）
- 验收标准：走查逐条有证据与修法；违反项有优先级；不停留在「感觉不好看」
- 复用范围：附录A 设计系统 · 各案例专属屏走查

## a11y-responsive

- 类型：验收·守护型
- 触发条件：当交付前要保无障碍与多端自适应、不能只在自己大屏上看着行时
- 输入：页面/组件 + 目标断点与无障碍等级（WCAG AA）
- 澄清问题：键盘能不能全程操作？对比度达标吗？窄屏塌不塌？语义标签对不对？
- PRD 片段：以语义标签 + 键盘可达 + WCAG AA 对比度 + 响应式断点（flex/grid + 相对单位）做交付前门禁，宽内容横向滚动不撑破 body（来源：WCAG 2.x + 响应式设计实践，核实 2026-07）
- 验收标准：Tab 可达全部交互；对比度≥AA；≤768px 不横向溢出；图有 alt、按钮有可读名
- 复用范围：附录A 设计系统 · 工作台前端（Next.js 响应式）
