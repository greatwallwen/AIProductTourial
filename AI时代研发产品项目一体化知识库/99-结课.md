## 结课 Capstone：一条跨案例的端到端任务

把「底座 → 检索 → 分层 → 经营方案」串成一次完整的产品闭环——这正是把 L0–L4 能力地图走一遍：

> **任务**：为一家零售企业做一份「会员经营改进方案」。
> 1. **建库（案例 45）**：把经营数据落成关系库，能按区域/品类秒查（真实 SQL + 索引）。
> 2. **检索（案例 44）**：用 RAG 从产品知识库里，只召回与「会员流失」最相关的几段方法，喂给分析。
> 3. **分层（案例 30）**：用 RFM 把会员真分层，揪出「过去高价值、近期久未乘机」的高价值流失群。
> 4. **成方案（案例 41）**：把结论写成「现状 → 问题 → 动作 → 责任人 → 验收标准」的可交付方案，每条都能被真实数据验证。
>
> **验收**：方案里每个数字都能溯源到真实数据；每个动作都有责任人和可执行的验收条件；凡是教学合成的列（如毛利率），都明确标注、不当真实结论。这就是本书反复强调的「诚信」与「可验收」。

## 结业与下一步

- **把方法用到自己的项目**：挑一个你手头真实的需求，走一遍「理念 → 原理 → 规范 → 设计 → 案例演示」——尤其别跳过「先建能验收的传感器，再谈自动化」。
- **深挖工程规范**：`rules/`（对齐 Google Style、SWE at Google、OWASP、12-Factor，出处见 `rules/references.md`）。
- **复用 Loop 实操**：`skills/loop_engineering/`（builder/checker/orchestrator/stop-rules/memory，工具无关，可套任意 Agent）。
- **记住一句话**：AI 时代产品经理的护城河，不是会不会写 prompt，而是懂不懂把「判断」沉淀成可复用、可验收的工程标准——并且，**永远诚实地区分真实与合成**。

到这里，你已经把一套「能跑、能改、能验收」的数字化产品方法走了一遍。去把它用起来。
